# Reglas de Auditoría de Código (GGA)

Esta es la constitución del proyecto para mantener un código limpio y libre de ruido.

## Calidad de Código

1. **Regla 1 (Zero Noise):** Rechazar el código si hay comentarios narrativos de IA (ej. `// Step 1: create node`, `// iterar el array`). El código debe ser auto-descriptivo.
2. **Regla 2 (JSDocs):** Exigir bloques JSDoc (`/** ... */`) formales para las interfaces y funciones exportadas.
3. **Regla 3 (El Porqué):** Solo se permiten comentarios en línea si explican el _porqué_ de una decisión de negocio, un workaround técnico o lógica compleja (como física de D3 o FTS5). Nunca el _qué_.
4. **Regla 4 (No `any`):** Rechazar el uso de `any` en TypeScript. Usar `unknown` con narrowing explícito, o definir el tipo correcto.

## Arquitectura (Inviolable)

5. **Regla 5 (Zero Frontend Data Filtering):** Rechazar código en `webview-ui/` que filtre, ordene o transforme datos del grafo (`Array.filter`, `Array.sort`, `Array.reduce` sobre `nodes` o `links`). Los filtros son responsabilidad exclusiva del backend SQL.
6. **Regla 6 (No SQL Inline):** Rechazar strings SQL (`SELECT`, `INSERT`, `WHERE`, `FROM`) fuera de `src/backend/repository/queries.ts`. Toda query debe pasar por los builders del Query Object pattern.
7. **Regla 7 (IPC Constants):** Rechazar strings literales de canales IPC en cualquier archivo que no sea `src/constants/ipc.ts`. Los mensajes deben usar siempre `IPC_CHANNELS.*`.
8. **Regla 8 (UI Primitivos):** Advertir si se crea un nuevo componente de botón, badge, o header en `webview-ui/src/components/` sin reutilizar o extender los primitivos de `webview-ui/src/components/ui/`.
